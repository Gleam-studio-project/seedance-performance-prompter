# 表演 Prompt 工作台 UI/UX 重构设计

## 1. 问题定义

当前页面把剧本、人物、人设确认、全局调教、模型配置和 Prompt 输出同时放在三列中。它看起来像“信息很多的控制台”，但不符合实际工作顺序：用户先要导入剧本，再定位场次，再确认人设，最后生成和检查 Prompt。

当前主要阻塞：

- 三列并行布局没有表达流程优先级，用户不知道第一步该做什么。
- 选择剧情依赖 textarea 的隐性文本选区，移动端尤其难用，也没有明确的“已选场次”对象。
- “AI 生成人设”“确认人设”“AI 生成 Prompt”“本地规则兜底”按钮并列，主次关系不清楚。
- 全局调教和模型/Base URL/API Key 位于主工作区，增加认知负担，也把高级配置暴露在常用路径上。
- 右侧 Prompt 输出是大文本框，格式自检只是底部列表，用户不容易判断哪里需要修改。
- 1180px 以下三列直接纵向堆叠，用户要反复滚动才能完成一次任务。
- 当前前端自检仍检查旧版负向约束文本，可能把已经合格的 Prompt 显示为警告。

## 2. 产品目标与成功标准

目标用户是短剧编剧、导演、制片和提示词编辑。他们的核心任务不是“管理模型”，而是把一段剧本快速变成可复制的表演 Prompt。

首版重构以以下指标验收：

- 新用户在 3 分钟内完成“导入剧本 -> 选择场次 -> 确认人设 -> 生成 Prompt”。
- 用户始终能看到当前步骤、已完成前置条件和下一步主按钮。
- 选择场次不依赖精确拖选；桌面和移动端都能通过点击完成。
- 生成 Prompt 最多需要一次主操作，复制结果最多两次点击。
- 任何异步请求都有加载、成功、失败和重试状态。
- 375px 宽度下不出现横向滚动，主要按钮触控区域不小于 44px。

## 3. 推荐信息架构

采用四步工作流，允许返回修改，但不强迫用户线性锁死：

1. **剧本**：导入/粘贴全文，定位并确认目标场次。
2. **人设**：查看 AI 草案，按人物卡片确认或修改表演档案。
3. **场次**：设置场景类型、时长、市场和可选表演偏好。
4. **Prompt**：生成、检查、复制或下载结果。

顶部只保留项目名、自动保存状态、新建和设置入口。模型/Base URL/API Key 放入“设置”抽屉；默认只显示模型连接状态，不把供应商配置混入创作流程。

主导航使用水平 Stepper：

```text
剧本  ->  人设  ->  场次  ->  Prompt
 1/4       2/4      3/4       4/4
```

每一步都有清晰的完成条件：

- 剧本：已有文本 + 已确认场次。
- 人设：至少一名人物档案已确认。
- 场次：时长和场景类型已确定。
- Prompt：生成结果通过格式检查。

## 4. 页面布局

### 4.1 桌面端

采用“左侧步骤导航 + 中央工作区 + 右侧上下文摘要”的布局，而不是三块等权面板：

- 左侧 216px：步骤、完成状态、项目最近保存时间。
- 中央：当前步骤的主要编辑区，占 1fr，宽度建议 720-900px。
- 右侧 280px：当前场景摘要、前置条件、下一步提示；Prompt 步骤时改为输出操作栏。
- 顶部 64px：项目名、市场标签、模型连接状态、自动保存、新建、设置。

### 4.2 各步骤布局

**剧本步骤**

- 上方是导入区：支持 PDF/DOCX/DOC/TXT/MD，显示文件名、字符数、解析状态和重试入口。
- 中间左侧是剧本编辑器，保留全文编辑能力；右侧是“场次列表/选段卡片”。
- 自动按标题、空行和对白段落生成候选场次；用户点击候选场次即可选中，不必拖 textarea。
- 底部固定“使用此场次，下一步”按钮；没有选段时明确显示禁用原因。
- 仍保留手动选择作为高级入口，选择后转换为可编辑的“当前场次”卡片。

**人设步骤**

- 顶部显示“AI 生成人设草案”和来源剧本摘要。
- 主体使用人物卡片，每张卡片展示人物名、性格底色、关系动力学、Tell、微表情和身体动作五组字段。
- 字段可以折叠，默认只展开需要确认的内容；文本编辑使用普通可读字体，不使用大块代码样式。
- 卡片右上显示“待确认/已确认”；底部主按钮是“确认人设并进入场次”。

**场次步骤**

- 顶部固定显示已选场次和人物摘要，避免用户来回滚动确认上下文。
- 主设置使用分组控件：场景类型、时长、市场；高级选项包含全局调教、物理环境和负向约束。
- “生成 Prompt”是唯一主按钮；本地规则兜底作为错误状态里的次级操作，不与主按钮并列。

**Prompt 步骤**

- 左侧展示结构化预览：镜头时间轴、每个镜头的主体/动作/环境/镜头/氛围字段。
- 右侧展示可复制的完整 Prompt，顶部提供复制、下载和重新生成。
- 格式检查改为顶部状态条：`通过 6/6` 或 `需要处理 2 项`，点击后定位到具体问题。
- 生成历史先做当前项目内的最近 5 个版本，后续再决定是否建设持久化版本库。

### 4.3 移动端

移动端切换为单列步骤流：顶部固定项目栏，下面是 Stepper，底部固定“返回/下一步/生成”操作栏。剧本编辑器和场次列表改为上下折叠区域；人设卡片逐张展开；Prompt 使用全屏阅读模式和固定复制按钮。禁止把桌面三列简单改成超长纵向页面。

## 5. 交互规则与状态

- 所有异步按钮在 300ms 后显示加载状态并禁止重复提交；文案使用“正在提取”“正在生成人设”“正在生成 Prompt”。
- 错误信息放在对应控件下方，必须告诉用户下一步，例如“模型不可用，点击重试或使用本地规则生成”。
- 生成前置条件使用可点击清单：点击“缺少人设确认”直接跳转到人设步骤。
- 自动保存显示为“已保存”“保存中”“保存失败，重试”，不能只依赖 Toast。
- 选段变更后立即使人设确认状态失效，并在 Stepper 上标记需要重新确认。
- 模型设置默认折叠；API Key 只在后端允许时显示，且提供显示/隐藏切换。
- 键盘用户可用 Tab 完成主要流程，焦点环清晰；错误使用 `aria-live`，不可只用颜色表达状态。

## 6. 视觉与组件规范

视觉方向为“编辑工作台”而非营销页：浅灰中性画布、白色编辑区、深墨文字、青绿色主色、橙色警告。去掉当前网格背景和大面积阴影，减少视觉噪音。

建议 tokens：

- 颜色：`--canvas #F4F6F8`、`--surface #FFFFFF`、`--ink #18212B`、`--muted #66717D`、`--accent #087F8C`、`--warning #B45309`、`--success #18794E`、`--border #D7DEE5`。
- 间距：4、8、12、16、24、32、40px。
- 字体：中文使用系统无衬线；结果 Prompt 使用等宽字体；标题只保留一档强调，不在每个面板重复大标题。
- 控件：按钮和输入高度 40-44px，卡片圆角 6px，边框优先于阴影；主按钮每个视图只保留一个。
- 图标：文件、设置、复制、下载、返回、成功、警告使用统一图标，并配合文字或 tooltip。

## 7. 分阶段实施计划

### P0：先修复可用性阻塞

- 修正前端格式自检与服务端当前 Prompt 契约不一致的问题。
- 增加顶部步骤状态和清晰的前置条件提示。
- 将模型配置移入设置抽屉，隐藏不必要的客户端 Key/Base URL 字段。
- 将“本地规则兜底”移入失败态，不再和 AI 主按钮竞争。

### P1：重排主流程

- 重构 `index.html` 为四步工作流结构。
- 在 `app.js` 增加 `currentStep`、`sceneSelection`、`saveStatus` 和 `generationStatus` 状态。
- 剧本步骤增加候选场次列表和“当前场次”对象，保留 textarea 作为编辑器。
- 人设步骤从大文本框改为人物卡片视图，同时保留原始文本编辑入口。

### P2：输出与响应式体验

- Prompt 步骤增加结构化镜头预览、复制反馈、版本切换和问题定位。
- 按 375px、768px、1280px 三个断点重做布局，不再使用“桌面三列然后全部纵向堆叠”。
- 增加浏览器自动化验收：导入示例、选场次、确认人设、生成、复制、失败重试。

### P3：真实用户验证

- 让 3-5 名真实用户完成各 3 次任务，观察首次点击位置、卡住步骤、人工修改量。
- 重点记录：完成时长、选段成功率、生成按钮误点率、返回修改次数和移动端完成率。
- 只有 P3 数据证明流程变快后，再考虑项目列表、云端共享和多人协作。

## 8. 文件级改动边界

- `index.html`：重组页面语义结构、步骤导航、场次列表、设置抽屉、输出操作区。
- `styles.css`：建立新的布局网格、响应式断点、状态和可访问性样式；删除三列纵向堆叠规则。
- `app.js`：增加步骤状态、场次选择、保存/生成状态、人物卡片渲染和错误定位。
- `tests/`：补充状态机和 Prompt 自检回归；后续加入浏览器流程测试。
- `Progress.md`：记录每一阶段验收数据，不把视觉主观评价写成完成标准。

## 9. 关键取舍

推荐先做 P0 + P1，而不是一次性增加项目列表、协作、拖拽时间线等功能。用户当前的主要损耗来自“找不到下一步”和“无法确认当前选段”，不是缺少更多能力。把工作流变短、把状态说清楚，收益会高于继续堆叠设置项。
